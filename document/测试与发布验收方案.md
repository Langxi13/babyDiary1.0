# Baby Diary 测试与发布验收方案

## 1. 目标与原则

本方案用于验证前端、后端、数据库迁移、Redis、AI 接口、PWA 壳、媒体链路和部署产物，避免只检查页面外观而遗漏接口 500、权限越界、迁移失败、缓存配置失效或移动端功能不可用。

执行原则：

1. 自动化测试只使用合成账号、合成日记和隔离基础设施，不复制生产用户数据、图片、数据库 dump 或 AI Key。
2. 先验证低成本静态规则和单元测试，再运行 MySQL 集成、三浏览器 E2E、预发布 DAST 和性能测试。
3. 所有发布都必须经过相同脚本，不以人工口头确认替代可重复的命令和报告。
4. 数据库迁移、认证、权限、媒体和删除操作属于高风险路径，必须有真实 MySQL 或端到端覆盖。
5. 自动化无法完整模拟 PWA 安装、系统分享和手机文件选择，发布前必须补 iPhone 与 Android 真机验收。

## 2. 风险与验证映射

| 风险区域 | 典型故障 | 主要验证 | 发布条件 |
| --- | --- | --- | --- |
| 数据库迁移 | 旧库升级失败、排序规则冲突、字段重复 | Testcontainers MySQL 8.4、旧结构基线迁移测试、预发布启动 | Flyway 到达最新版本且无迁移错误 |
| 相册与媒体 | 相册组 500、图片路径匹配失败、分页或缩略图失效 | 后端集成测试、三浏览器相册路由、k6 相册接口、生产冒烟 | 相册接口无 5xx，首屏图片可加载 |
| 认证与权限 | 会话提前失效、跨用户读取、锁定日记泄漏 | 服务测试、真实 HTTP 集成测试、E2E 登录、权限负例 | 未授权响应正确，数据边界无泄漏 |
| 日记主流程 | 创建、编辑、图片顺序、详情、搜索回归 | Node/Vitest、真实 HTTP 集成、E2E 路由、真机上传 | 创建到详情的完整链路可用 |
| AI 报告 | 模型列表失败、超时、提示词视角错误 | WireMock AI E2E、服务测试、可选真实提供商冒烟 | Mock 必过；启用 AI 时真实冒烟通过 |
| Redis | 配置前缀失效、容器内误连 localhost、缓存故障拖垮接口 | 预发布健康检查、Redis E2E、缓存降级测试 | Redis 健康，故障时业务可回源数据库 |
| 桌面与手机 UI | 导航遮挡、日期弹窗溢出、文本重叠 | Playwright 多视口、视觉快照、axe、真机 | 无关键遮挡和严重无障碍问题 |
| PWA 壳 | 安装、分享目标、文件选择、图标异常 | Manifest/Service Worker 测试、Android/iPhone 真机 | 安装后可启动，支持的平台链路可用 |
| 部署与安全 | root 容器、缺安全头、伪健康 200、临时目录权限破坏、凭据误提交 | Trivy、Gitleaks、ZAP、脚本治理、真实 Actuator 健康检查 | 高危/严重项为 0，ZAP 阻断项为 0，Actuator 顶层状态为 UP |

## 3. 测试环境分层

### 3.1 本地快速验证

- JDK 17、Maven 3.9+、Node.js 22。
- Shell 测试验证备份、权限、健康检查、数据库查询说明和验证脚本。
- 前端 Node 测试覆盖静态契约、响应式样式、PWA 元数据和关键交互规则。
- Vitest + jsdom 覆盖导航和相册组件行为。

### 3.2 后端集成环境

- Testcontainers 启动 MySQL 8.4，不使用 H2 替代生产 SQL 行为。
- 从空库执行 V1-V14，并从合成旧结构基线执行 V14 升级。
- 通过随机 HTTP 端口验证注册、登录、日记、相册、空间与权限响应。

### 3.3 浏览器 E2E 环境

`compose.e2e.yaml` 提供：

- MySQL 8.4 临时数据目录。
- Redis 7.4。
- WireMock AI 提供商，固定返回模型列表和报告内容。
- 可选 Mailpit 与 MinIO，不连接真实邮件、对象存储或 AI 服务。

Playwright 运行 Chromium、Firefox 和 WebKit。axe 与视觉快照固定在 Chromium 执行，业务路由和 AI Mock 在三个浏览器执行。

### 3.4 打包预发布环境

`compose.staging.yaml` 使用与发布一致的前后端 Dockerfile，并按 2 核 4GB 服务器约束资源：

- MySQL 1GB / 0.75 CPU。
- Redis 192MB / 0.25 CPU，最大缓存 128MB。
- 后端 1536MB / 0.8 CPU。
- 前端 128MB / 0.2 CPU，以非 root Nginx 运行。

预发布只使用随机密钥和合成数据，用于 ZAP、k6、容器健康检查和安全头验证。

### 3.5 生产冒烟

生产只做只读或可清理的最小验证：健康检查、登录页、Manifest、未登录认证响应和已授权核心列表。禁止在生产自动创建大量账号、日记或媒体。

## 4. 自动化测试矩阵

### 4.1 脚本治理

```bash
scripts/verify-scripts.sh
```

要求所有 `scripts/*.test.sh` 通过，并检查 Shell/JavaScript 语法、Compose 展开和关键运行目录权限。健康检查必须拒绝 Actuator `DOWN` 或 SPA HTML 伪响应；部署必须先通过 `nginx -t` 再停止后端；systemd 必须启用 `PrivateTmp`，且图片权限脚本不得修改 `/tmp` 等共享系统目录。所有外部 GitHub Actions 必须固定到 40 位完整提交 SHA。安全扫描快照必须正确处理“Git 已跟踪但工作区已删除”的文件。

### 4.2 后端测试与覆盖率

```bash
scripts/verify-backend.sh
```

当前覆盖内容包括：

- 服务、校验器、JWT、缓存降级、文件存储、AI 客户端和 Mapper SQL/XML 测试。
- MySQL 8.4 全栈接口测试。
- V13 旧库排序规则迁移测试。
- Java 17、Spring Boot 3.5、安全配置、隐私派生和模块边界基线。

JaCoCo 硬门槛：

- 行覆盖率不低于 45%。
- 分支覆盖率不低于 28%。

覆盖率报告位于 `backend/target/site/jacoco/`。

### 4.3 前端测试与覆盖率

```bash
scripts/verify-frontend.sh
```

该脚本执行所有 `public/`、`src/` 下的 `*.test.js`，随后运行 Vitest 覆盖率和 Vite 生产构建。

Vitest 硬门槛：

- Lines 50%。
- Statements 45%。
- Functions 30%。
- Branches 30%。

当前组件覆盖重点为顶部导航、相册失败重试与相册分组行为。后续修改高风险页面时，应把相应组件纳入 `vite.config.js` 的覆盖范围，而不是通过降低门槛解决失败。

### 4.4 三浏览器 E2E

```bash
scripts/verify-e2e.sh
```

验证内容：

- AI 模型列表与周报生成只访问 WireMock。
- 桌面导航在 1366px 完整显示，高风险临界宽度自动切换紧凑导航。
- 使用两个合成账号验证 A 缓存纪念日后退出并登录 B，无需刷新页面即可看到 B 的空状态，且 Chromium、Firefox、WebKit 均不得短暂显示 A 的内容。
- 首页、日记、共同空间、时间轴、日历、纪念日、相册、AI 报告、草稿、个人信息、通知和写日记路由无浏览器错误或后端 5xx。
- 首页、日记、相册和个人信息页无 serious/critical axe 问题。
- 经过人工确认的桌面与手机视觉快照无漂移。

失败产物位于 `frontend/test-results/`，HTML 报告位于 `frontend/playwright-report/`。不要提交其中的登录状态、截图或 trace。

### 4.5 性能验证

```bash
STAGING_PROJECT_NAME=baby-diary-performance scripts/start-staging.sh
scripts/run-k6.sh
STAGING_PROJECT_NAME=baby-diary-performance scripts/stop-staging.sh
```

默认场景创建 50 个合成用户，峰值 10 个并发用户，持续 3 分钟循环读取空间、V2 日记、旧版日记、相册组和所有照片分页。

门槛：

- 检查成功率大于 99%。
- HTTP 失败率小于 1%。
- P95 小于 800ms。
- P99 小于 1500ms。
- HTTP 5xx 比例必须为 0。

性能工作流每周运行，也可在 GitHub Actions 手动调整稳态时长。合并共享查询、数据库索引、相册或缓存改动前应手动运行。

### 4.6 安全验证

```bash
scripts/security-scan.sh
```

包含：

- `npm audit --omit=dev`：高危依赖阻断。
- Trivy：Maven/NPM 依赖、源码密钥和 Dockerfile 高危/严重配置阻断。
- 公开 refs 获取：拉取远端分支、标签、Notes、PR head 和 PR merge 引用，覆盖删除源分支后仍可公开访问的提交。
- 隐私披露守卫：扫描当前非忽略文件、提交/标签元数据和全部可达 Git 历史，阻断未批准域名、邮箱、IP、服务器路径、个人标识和敏感文件名，并仅输出类别与位置。
- 公开资产校验：图片、文档、归档、字体和音视频必须经过人工内容/元数据检查，其当前及历史 SHA-256 必须存在于 `config/public-asset-allowlist.sha256`。
- Gitleaks：同时扫描当前源码快照和全部可达 Git 历史；测试密钥也必须使用明确的低熵 `test-only` 值。

允许公开的示例域名及公共依赖主机只在 `config/privacy-host-allowlist.txt` 中维护。新增条目必须经过人工确认，生产域名、个人网站和自托管地址不得加入允许列表。

打包预发布后执行：

```bash
scripts/run-zap-baseline.sh
```

ZAP 对缺少 CSP、`X-Content-Type-Options`、点击劫持保护、跨域错误和错误详情泄漏等规则设为阻断。打包注释、SPA 识别和当前确有必要的 `style-src 'unsafe-inline'` 可记录为非阻断告警，但不得隐藏新的高风险项。

## 5. 功能验收清单

### 5.1 认证与账号

1. 注册、登录、刷新、退出和 30 天刷新会话正常。
2. 错误密码、失效刷新令牌和未授权接口返回正确状态，不出现 500。
3. Web 与手机端头像显示一致，头像拖拽/点击上传后重新登录仍存在。
4. 个人信息入口可点击，窄屏下标签、输入框和操作按钮不重叠。
5. 修改密码后旧设备会话按设计失效。
6. 账号 A 退出并登录账号 B 后，API 缓存、日记/空间 Store、当前空间和二次验证状态立即清理；无需手动刷新，B 不得看到 A 的纪念日或其他页面数据，A 的延迟响应也不得回写。
7. 仅管理员能看到邀请码管理区；查看和刷新必须完成密码二次验证，普通用户接口返回 `403`，缺少二次验证返回 `423`。
8. 数据库只保存 `v1:` AES-GCM 密文；刷新后旧邀请码注册失败、新邀请码注册成功，明文响应带 `Cache-Control: no-store` 且不会写入前端持久缓存。

### 5.2 日记与草稿

1. “写日记”默认创建新日记且日期为今天，只有草稿入口加载指定草稿。
2. 标题、正文、Emoji 心情、默认标签颜色、日期、图片上传和图片排序可保存。
3. 列表预览超过阈值后省略，点击卡片可进入完整详情。
4. 搜索、起止日期、分页、时间轴和日历在桌面与手机均不溢出。
5. 编辑、删除、回收站、恢复、锁定和权限负例符合数据边界。

### 5.3 相册与媒体

1. `/api/albums/groups` 在空库、包含旧图片路径和大量图片时均返回成功。
2. “所有图片”“收藏照片”、年份、AI 和自建相册封面使用已有图片，不显示文字占位。
3. 相册首页只展示相册卡片，点击进入详情后按 24 张分页加载。
4. 缩略图失败有明确状态，组请求失败可在页面内重试。
5. iPhone Safari/PWA 可显示同源图片；Android PWA 的系统分享目标与浏览器上传兜底按支持情况工作。

### 5.4 AI 报告

1. 管理员保存 OpenAI 兼容配置后可加载模型列表和测试连接。
2. 周报、月报、年报的 Markdown 标题、列表和强调样式正确。
3. 报告使用第三方观察视角，可使用“你/你们”，不以模型第一人称冒充用户。
4. 锁定日记不进入 AI 输入，API Key 不回显。
5. 超时、提供商 4xx/5xx 和网络失败显示可理解错误，不把下游敏感详情返回前端。

### 5.5 导出、同步与协作

1. ZIP、PDF、EPUB 和图片导出使用合成数据验证，临时文件会清理。
2. 离线创建、编辑、删除、重复推送和冲突提示符合幂等规则。
3. 共同空间邀请、角色、评论、回应和通知只对有权限成员可见。
4. 最后一名所有者不能被并发删除或降级。

## 7. 发布门禁

以下条件全部满足才允许部署：

1. `scripts/verify-scripts.sh`、`scripts/verify-backend.sh`、`scripts/verify-frontend.sh` 全部通过。
2. 三浏览器 E2E 业务用例全部通过；只允许明确限定到 Chromium 的 axe/视觉用例在其他浏览器显示为 skip。
3. JaCoCo 与 Vitest 覆盖率达到硬门槛，不允许临时降低。
4. npm audit、Trivy、Gitleaks 无高危/严重阻断项。
5. 打包预发布后端、前端、MySQL 和 Redis 全部 healthy，ZAP 阻断项为 0。
6. k6 无 5xx，并满足 P95/P99 与失败率门槛。
7. 涉及 PWA、上传、媒体或响应式布局时，iPhone 与 Android 真机通过。
8. Flyway 发布前备份已完成，旧库升级测试与回滚路径已确认。
9. README、功能、架构、部署和 Changelog 已同步。
10. 主机 `/tmp` 为 `1777`、后端仅绑定回环地址、Nginx 精确代理 Actuator，公网健康响应顶层为 `UP`。
11. GitHub Actions 第三方步骤固定到完整提交 SHA，Dependabot 不存在未处理的高危或严重级供应链告警。

任何 500、数据越权、迁移失败、凭据泄漏、无法登录、无法查看日记或相册的问题均为发布阻断项。

## 8. 标准发布流程

```bash
npm --prefix frontend ci
scripts/verify.sh
scripts/verify-e2e.sh
scripts/security-scan.sh
```

使用 `config/staging.env.example` 创建私有 `.env.staging` 后：

```bash
scripts/start-staging.sh
scripts/run-zap-baseline.sh
scripts/run-k6.sh
scripts/stop-staging.sh
```

生产发布：

```bash
scripts/backup.sh
scripts/deploy.sh
scripts/health-check.sh
```

部署后检查后端日志中是否出现新的 `ERROR`、未预期 `WARN`、Flyway 失败或 HTTP 5xx，并确认公网 `/actuator/health` 返回真实 JSON `UP`、后端端口只绑定回环地址、systemd `PrivateTmp` 生效，再人工打开首页、日记、相册和个人信息页。

## 9. 失败分级与回滚

- P0：数据泄漏、数据损坏、认证绕过、无法回滚的迁移。立即停止发布并恢复备份。
- P1：登录、日记、相册等核心链路不可用或持续 5xx。回滚应用版本；若涉及迁移，按已验证方案恢复数据库。
- P2：单个次级功能不可用、明显布局遮挡、性能门槛失败。发布前修复，不带已知回归上线。
- P3：不影响操作的轻微样式偏差或非阻断扫描提示。记录问题并明确处理版本。

回滚时必须同时核对应用 Jar、前端静态文件、数据库版本、图片/对象目录和私有环境配置。不要只回滚前端而保留不兼容的后端或数据库结构。
