## 系统架构概述

### 整体架构

Baby-Diary系统采用前后端分离架构，后端只提供RESTful API服务，前端由Nginx托管Vue/Vite构建产物。

```
┌─────────────────┐         HTTP/HTTPS         ┌─────────────────┐
│   前端应用      │ ◄─────────────────────────► │   后端服务      │
│  (Vue 3 + Vite) │                             │ (Spring Boot)   │
└─────────────────┘                             └─────────────────┘
         │                                               │
         ▼                                               ▼
┌─────────────────┐                             ┌─────────────────┐
│   浏览器        │                             │   MySQL 数据库   │
│  (用户界面)     │                             └─────────────────┘
└─────────────────┘                                      │
                                                        ▼
                                               ┌─────────────────┐
                                               │ Redis 缓存      │
                                               └─────────────────┘
```

### 架构特点

1. **前后端分离**：前端和后端完全独立开发、部署和维护
2. **RESTful API**：后端提供标准的RESTful接口
3. **无状态认证**：使用JWT进行身份认证，服务端不保存会话状态
4. **分层设计**：后端采用经典的三层架构（Controller-Service-Mapper）
5. **REST-only**：旧Thymeleaf页面和后端模板Controller已移除
6. **移动端 App Shell**：Vue 前端在手机宽度下由全局 Shell 统一提供顶部栏、底部主导航和安全区布局，页面只负责业务内容
7. **安卓系统分享入口**：安装后的 PWA 通过 Web App Manifest `share_target` 注册为图片分享目标，安卓用户可从系统相册把图片分享到 Baby Diary，再进入写日记页继续编辑

### 构建与拆包

前端页面通过 Vue Router 懒加载。日记编辑页使用 Tiptap，相关 ProseMirror 代码只进入 `DiaryForm` 路由 chunk，不进入首页首屏预加载。Element Plus 组件按页面局部引入，避免全量 UI 库进入主包。桌面导航 hover、手机导航 touchstart 时会预加载目标路由 chunk，降低首次切换页面的等待时间。

前端 API 层提供内存 TTL 缓存和同 key 请求去重。列表、首页、相册和草稿默认缓存 30 秒，时间轴和日历默认缓存 2 分钟，标签和纪念日默认缓存 10 分钟；创建、更新、删除、收藏等写入操作会失效相关缓存。缓存使用 generation 标记隔离失效前的在途请求，旧请求完成后不能重新写回过期数据，也不能误删新的在途请求。

页面级 SCSS 从 `DiaryForm`、`AiReports`、`Anniversaries`、`Album`、`AlbumDetail`、`DiaryList`、`Timeline`、`Home` 和移动壳组件中拆出到独立样式文件。日记图片选择、安卓上传兜底和 PWA 分享导入集中在 `useDiaryImages` composable，降低单个 SFC 的维护规模。全局按钮和页面过渡只声明实际变化的 CSS 属性，不使用 `transition: all` 触发布局无关动画。

`frontend/src/assets/styles/mobile-foundation.scss` 作为手机端公共布局层，集中定义顶部栏、底部栏、页面边距、表面颜色和控件圆角变量。在 `768px` 及以下统一提供安全区内边距、最小 `320px` 布局基线、横向溢出保护、输入控件 16px 字号与至少 44px 触控高度，并约束 Element Plus 的对话框、消息、确认框、日期面板和下拉面板不超出动态视口。页面样式只保留各业务区域自身的网格和视觉层级，减少重复的移动端基础规则。

### PWA 与移动壳能力

`frontend/public/manifest.webmanifest` 提供安装元数据，并通过 `share_target` 注册 `/share-target` 作为安卓系统图片分享入口。`frontend/public/sw.js` 拦截该入口的 `POST multipart/form-data` 请求，只接受 10MB 内的 JPEG、PNG、GIF、WebP 文件，写入专用 Cache API 命名空间 `baby-diary-share-target-v1`，再重定向到 `/diaries/create?shared=1`。普通 GET 只缓存同源成功响应；网络失败时只有页面导航请求回退到 `index.html`，静态资源失败不会再错误返回 HTML。写日记页通过 `shareTargetFiles.js` 消费缓存文件，最终仍走创建日记接口的 `imageFiles` 字段。

`MobileAppShell` 将手机端顶部栏、五项主导航和次级功能抽屉与桌面 `NavBar` 分离。主导航和抽屉入口在触摸开始时预加载路由 chunk；抽屉打开时锁定页面滚动并支持 Escape 关闭。壳层监听 `visualViewport` 高度变化，识别软键盘后隐藏底部导航，避免导航遮挡输入区。日记列表在手机端使用两个独立的单日期选择器，避免双日历面板在窄屏并排溢出。

该能力依赖浏览器对 Web Share Target 的支持，主要面向安装后的 Android Chrome PWA。安卓壳内原生文件选择器仍保留浏览器上传链接兜底；iOS 和普通浏览器继续使用原有文件选择入口。

生产敏感配置不写入仓库。后端默认配置位于 `backend/src/main/resources/application.yml`，生产覆盖配置位于 `config/application-prod.yml`，真实密钥和数据库连接从 `/etc/baby-diary/backend.env` 注入。生产 Java 进程使用专用系统用户 `baby-diary` 运行，数据库连接使用业务账号 `baby_diary_app`，不使用 root 作为应用运行账号。

### 查询与缓存优化

- 日记列表支持 `summary=true`，只读取 `LEFT(content, 512)` 作为预览，详情和编辑接口仍读取完整正文。
- 日记列表、时间轴和草稿列表只读取正文摘要；详情、编辑和单份草稿接口仍读取完整正文。
- 日记列表和时间轴按当前结果 diaryId 批量查询标签和图片，避免每篇日记单独查询。
- Flyway V4 增加 `diary(user_id,date,created_at)`、`diary(user_id,mood_key,date)`、`diary_image(diary_id,sort,image_id)` 和 `diary_tag(tag_id,diary_id)` 索引。
- Flyway V5 增加 `ai_config` 和 `ai_report`，用于保存 OpenAI 兼容配置和周报/月报历史。
- Flyway V7 增加 `album_photo(album_id,sort,image_id)` 和 `ai_report(user_id,created_at,report_id)`，覆盖相册详情和 AI 报告历史排序查询；相册详情排序收敛为 `album_photo.sort,image_id`，避免跨表排序 filesort。
- Flyway V8 将 `diary.content` 扩展为 `MEDIUMTEXT`，与富文本日记和草稿容量保持一致。
- 通用照片、系统相册和自建相册详情提供数据库 `LIMIT/OFFSET` 分页；相册首批默认24张，元数据和首批照片并行请求，首页收藏预览只读取6张，避免照片量增长后一次返回和解码全部图片。
- 生产环境启用 Spring Cache + Redis，缓存命名空间前缀为 `baby-diary:prod:`，读缓存 key 均包含 userId 或用户作用域参数。
- Redis 读写或清理异常由 `CacheErrorHandler` 降级处理，业务查询继续回源数据库，避免缓存服务成为页面可用性的单点。
- 缓存名称统一由 `CacheNames` 管理：`diaryPage` 30秒、`diaryTimeline` 2分钟、`diaryCalendar` 2分钟、`tags` 10分钟、`anniversaries` 10分钟、`photos` 30秒、`drafts` 30秒。
- 相册组摘要、系统照片分页和自建相册分页复用 `photos` 缓存；日记图片、收藏和相册写操作统一清空该缓存。
- 时间轴和日历查询使用日期范围条件，不在 WHERE 中使用 `YEAR(date)` 或 `MONTH(date)`，以便复用日期索引。
- 相册封面回退策略不再使用 `ORDER BY RAND()`，改为按相册照片排序稳定选择已有图片；后续数据规模继续增大时可扩展为持久化 `cover_image_id`。
- 前端缩略图 URL 统一由 `thumbnailImageUrl()` 生成，列表、相册、首页、时间轴等小图区域读取 `/images/thumbs/480/`，全屏预览和导出继续读取原图，降低首屏图片传输体积。
- 日记列表 Store 使用递增请求序号，只允许最后一次筛选请求更新页面；关键词输入使用350毫秒防抖，减少无效查询并避免旧响应覆盖新筛选。
- `PageResult.map()` 统一分页实体到 VO 的转换，照片和 AI 报告控制器不再重复组装分页元数据。
- `/api/**` 注册慢请求日志拦截器，生产默认阈值 `SLOW_REQUEST_THRESHOLD_MS=1000` 毫秒。

## 系统分层架构

### 后端分层

```
┌─────────────────────────────────────────────────┐
│                   Controller层                    │
│  (处理HTTP请求，参数校验，返回响应)               │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│                    Service层                      │
│  (业务逻辑处理，事务管理)                         │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│                   Mapper层                       │
│  (数据访问，SQL执行)                              │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│                   MySQL数据库                     │
└─────────────────────────────────────────────────┘
```

### 层职责说明

#### Controller层

- 接收HTTP请求
- 参数校验和绑定
- 调用Service层处理业务
- 封装响应结果返回

主要Controller：
- `AuthController`：认证相关接口
- `DiaryApiController`：日记管理接口
- `PhotoController`：图片管理接口
- `AlbumController`：相册和相册组接口
- `AiController`：AI 配置和周报/月报接口

#### Service层

- 业务逻辑处理
- 事务管理
- 调用Mapper层访问数据
- 处理业务规则

主要Service：
- `LoginService`：用户认证服务
- `DiaryService`：日记管理服务
- `DiaryImageService`：日记图片服务
- `DiaryArchiveService`：图片 ZIP 导出和临时文件生命周期
- `ImageStorageService`：统一图片校验、随机命名、缩略图和事务清理
- `AiConfigService`：AI 配置读取、保存和连接测试
- `AiReportService`：报告周期解析、日记聚合、AI 调用和报告保存

#### Mapper层

- 数据库操作
- SQL定义和执行
- 数据映射

主要Mapper：
- `UserMapper`：用户数据访问
- `DiaryMapper`：日记数据访问
- `DiaryImageMapper`：日记图片数据访问
- `AiConfigMapper`：AI 配置数据访问
- `AiReportMapper`：AI 报告历史数据访问

### 后端模块边界

当前阶段先治理边界和依赖方向，不做大规模搬包，避免影响已上线功能。后续重构可以按以下模块拆包：

| 模块 | 当前主要类 | 边界约束 |
|------|------------|----------|
| auth | `AuthController`、`LoginService`、`JwtTokenProvider`、`JwtAuthenticationFilter`、`CurrentUser` | 只负责认证、JWT 和当前用户识别，不直接处理日记、相册或 AI 业务 |
| diary | `DiaryApiController`、`DiaryDraftController`、`TagController`、`DiaryService`、`DiaryArchiveService`、`ImageStorageService`、`DiaryDraftService`、`TagService`、`DiaryMapper` | 日记正文、草稿、标签、时间轴、日历、导出和图片生命周期 |
| album | `AlbumController`、`PhotoController`、`AlbumService`、`PhotoService`、`AiAlbumProposalService`、`AlbumMapper`、`PhotoMapper` | 相册组、相册照片索引、系统相册和 AI 相册确认流程 |
| ai | `AiController`、`AiConfigService`、`AiReportService`、`OpenAiCompatibleClient`、`AiConfigMapper`、`AiReportMapper` | AI 配置、模型列表、报告生成和报告历史；不直接返回明文 API Key |
| experience | `AnniversaryController`、`AnniversaryService`、`AnniversaryMapper` | 纪念日和扩展体验功能 |
| common | `Result`、`PageResult`、`Pagination`、`ErrorCode`、`CacheNames`、`GlobalExceptionHandler`、`HtmlSanitizer`、`DiaryRequestValidator` | 统一响应、分页上限、错误码、缓存命名、输入边界和跨模块基础能力 |

接口响应继续统一为 `Result<T>`，分页响应统一为 `PageResult<T>`。新增接口应优先复用这两个结构，不再返回临时 Map 或裸数组作为顶层响应。

## 核心模块设计

### 认证模块

#### JWT认证流程

```
1. 用户提交用户名和密码
2. Spring Security验证用户名和密码
3. 验证成功后生成JWT Token
4. Token返回给前端
5. 前端在后续请求Header中携带Token
6. JwtAuthenticationFilter拦截请求并验证Token
7. 验证通过后设置SecurityContext
8. 放行请求到Controller
```

#### 关键类

- `JwtTokenProvider`：Token生成和验证
- `JwtAuthenticationFilter`：Token拦截和验证
- `SecurityConfig`：安全配置
- `CustomUserDetails`：用户详情实现

### 日记管理模块

#### 创建日记流程

```
1. 前端提交日记数据和图片
2. DiaryApiController接收请求
3. 调用DiaryService.saveDiary()
4. 保存日记基本信息到diary表
5. 获取生成的diaryId
6. 遍历图片文件，压缩处理
7. 保存图片文件到本地
8. 生成 `thumbs/480/{filename}` 缩略图
9. 保存图片信息到diary_image表
10. 返回创建结果
```

#### 图片压缩

使用Thumbnailator库进行图片压缩：

- 压缩阈值：300KB
- 最大尺寸：1920px
- 压缩质量：0.85

关键类：`ImageCompressor`

#### 图片缩略图

后端为每张上传图片维护 480px 缩略图：

- 存储位置：`${DIARY_FILE_PATH}/thumbs/480/`
- 访问路径：`/images/thumbs/480/{filename}`
- 生成策略：创建或追加上传图片后生成缩略图，小图不强制放大
- 清理策略：删除日记或编辑时移除旧图片，会同步删除对应缩略图
- 历史数据：`scripts/generate-thumbnails.sh` 会扫描 `data/images/` 的原图并补齐缺失缩略图

### 图片管理模块

#### 图片存储

- 存储位置：`DIARY_FILE_PATH` 指定的目录
- 缩略图位置：`${DIARY_FILE_PATH}/thumbs/480/`
- 文件名格式：`diary_{userId}_{uuid}.jpg`、`avatar_{userId}_{uuid}.jpg`、`anniversary_{userId}_{uuid}.jpg` 等随机服务端名称
- 支持格式：JPEG、PNG、GIF、WebP
- 文件大小限制：10MB
- 校验方式：同时校验允许的 MIME 和文件签名，不接受 SVG 或伪造图片类型
- 事务规则：数据库回滚时清理本次新文件，旧图片只在事务提交后删除，避免数据库与文件系统状态相反
- 导出规则：ZIP 临时文件在响应流关闭后自动删除

### AI报告模块

#### 生成流程

```
1. 前端选择周报或月报周期
2. AiController 校验登录用户和请求参数
3. AiReportService 解析周期，并查询当前用户该周期内的日记
4. 批量补充日记标签，仅整理日期、标题、文字内容、心情和标签
5. OpenAiCompatibleClient 调用配置的 /chat/completions 接口
6. 生成 Markdown 报告并写入 ai_report
7. 前端展示报告，并可在历史报告中再次查看或删除
```

#### 配置方式

- 使用 OpenAI 兼容 Chat Completions 风格配置：`baseUrl`、`model`、`apiKey`、`timeoutSeconds`。
- `baseUrl` 填写到版本路径，例如 `https://api.openai.com/v1`，后端自动拼接 `/chat/completions`。
- `apiKey` 只在保存时提交明文，后端使用 `AI_CONFIG_ENCRYPTION_KEY` 派生 AES-GCM 密钥加密后存库，读取配置时只返回脱敏状态。
- AI 输入不包含图片文件、图片地址或图片识别内容。

## 安全设计

### 认证安全

1. **密码加密**：使用Spring Security的BCrypt加密算法
2. **JWT Token**：
   - 使用HMAC-SHA256签名算法
   - Token默认有效期30天
   - 包含用户ID和用户名信息
3. **邀请码注册**：防止未授权注册

### 接口安全

1. **CORS配置**：生产环境只允许 `CORS_ALLOWED_ORIGINS` 中明确配置的来源
2. **权限控制**：用户只能访问和操作自己的数据
3. **参数校验**：使用 Validation 和 `DiaryRequestValidator` 校验日期、长度、格式、图片数量与分页上限；分页每页最多100条
4. **SQL注入防护**：使用MyBatis参数化查询
5. **健康检查最小暴露**：Actuator 仅开放 `/actuator/health`，用于部署后探活和运维检查

### 文件上传安全

1. **文件类型校验**：只允许 JPEG、PNG、GIF、WebP，并校验真实文件签名
2. **文件大小限制**：单文件最大10MB
3. **文件名安全**：使用 UUID 随机文件名，不保留用户名和客户端原始文件名；所有路径都限制在上传根目录内
4. **目录权限**：上传目录属主为 `baby-diary`，属组为 `www-data`，目录权限为 `2750`，图片文件权限为 `640`；后端负责写入，Nginx 只读取，避免 Java 进程以 root 写入业务文件，也避免静态图片因 Nginx 无权读取而返回 404
5. **资源归属**：手动相册和 AI 相册确认前都会重新校验 imageId 属于当前用户

### AI配置安全

1. **密钥加密存储**：API Key 使用 AES-GCM 加密后写入 `ai_config.encrypted_api_key`
2. **环境密钥注入**：生产加密主密钥来自 `/etc/baby-diary/backend.env` 的 `AI_CONFIG_ENCRYPTION_KEY`
3. **前端不回显明文**：配置读取接口只返回是否已配置和脱敏占位
4. **最小化发送内容**：报告生成只发送必要日记文字元数据，不发送图片内容
5. **连接地址约束**：Base URL 只接受 HTTP/HTTPS 且不能包含账号、查询参数或片段；下游异常详情只写入受控日志，不直接回传前端

### 错误处理

- 统一异常处理：`GlobalExceptionHandler`
- 业务异常：`BusinessException`
- 错误码：`ErrorCode`枚举
