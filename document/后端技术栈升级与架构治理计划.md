## 固定验证入口

每次后端升级、部署脚本调整或架构治理前后，统一运行：

```bash
cd /opt/baby-diary
scripts/verify.sh
scripts/backup.sh
scripts/deploy.sh
scripts/health-check.sh
scripts/runtime-governance-check.sh
scripts/explain-key-queries.sh
```

关键约束：

- `scripts/verify.sh` 使用后端 `mvn -q clean test`，避免 Maven 增量编译复用旧 `.class`。
- `scripts/deploy.sh` 使用后端 `mvn -q -DskipTests clean package -f backend/pom.xml`。
- Maven Compiler 必须保留 `<parameters>true</parameters>`，否则 Spring 6 无法稳定解析未显式命名的 `@RequestParam` 和缓存 SpEL 参数名。
- 部署前必须执行 `scripts/backup.sh`。
- 部署后至少检查核心页面、`/actuator/health` 和未登录 `/api/auth/info` 的 `401`。

## 已落地治理项

- Java 运行时和编译目标统一为 Java 17。
- Spring Boot 迁移到 3.5.16，`javax.servlet`、`javax.validation`、`javax.annotation.PostConstruct` 已迁移到 `jakarta.*`。
- Spring Security 改为 `SecurityFilterChain`、`@EnableMethodSecurity`、`requestMatchers`。
- springdoc 使用 `springdoc-openapi-starter-webmvc-ui` 2.x。
- MyBatis Spring Boot Starter 升级到 3.0.5。
- 后端使用专用系统用户 `baby-diary` 运行。
- 生产数据库连接使用业务账号 `baby_diary_app`。
- `/etc/baby-diary/backend.env` 保持 `600` 权限，敏感配置不进入仓库。
- Actuator 只暴露 `/actuator/health`。
- Redis Cache 已接入生产读缓存，缓存名称和 TTL 由 `CacheNames` 集中管理。
- Redis 缓存异常支持降级回源数据库，不再将 Redis 短暂故障放大为业务接口失败。
- 图片写入统一由 `ImageStorageService` 处理真实格式校验、UUID 命名、缩略图和事务清理。
- 日记请求边界由 `DiaryRequestValidator` 集中校验，分页统一限制每页最多100条。
- 日记列表、时间轴、日历、相册详情、AI 报告历史等关键查询已补索引和 EXPLAIN 诊断脚本。
- 相册封面回退策略移除 `ORDER BY RAND()`，改为稳定选择已有图片。
- `/api/**` 已接入慢请求日志，生产默认阈值 1000ms。

## 架构边界

当前代码仍保持现有包结构为主，不做大规模搬包。后续重构应按以下业务边界推进：

- `auth`：登录、注册、JWT、用户资料、头像、密码。
- `diary`：日记正文、图片、标签、心情、导出。
- `album`：系统相册、自建相册、相册组、相册照片。
- `ai`：AI 配置、模型列表、周报/月报、AI 相册推荐。
- `experience`：纪念日、草稿、首页聚合等体验增强能力。
- `common`：响应结构、错误码、异常处理、缓存名称、工具类。

治理原则：

- Controller 不直接访问 Mapper。
- Service 内部封装缓存失效和事务边界。
- Mapper XML 避免在索引列上使用 `YEAR()`、`MONTH()` 等函数过滤。
- 新增接口同步更新 API 文档和功能文档。
- 新增缓存必须明确 TTL、key 组成和失效入口。

## 数据库迁移基线

当前 Flyway 最新版本为 V8：

- V1：旧版基础表结构。
- V2：标签、心情、纪念日、相册收藏和草稿。
- V3：头像和 token 版本。
- V4：列表、筛选、图片排序和标签筛选索引。
- V5：AI 配置和周报/月报报告历史。
- V6：智能相册、相册组、相册照片、AI 相册提案。
- V7：架构治理索引。
- V8：日记正文扩展为 `MEDIUMTEXT`。

生产迁移必须通过 Flyway，不手工改业务表结构。需要回滚时优先恢复部署前备份，不在生产库直接逆向执行未验证 SQL。

## 后续升级原则

- 每次升级只处理一个主变量，例如 JDK、Spring Boot、数据库驱动或前端构建工具，不和业务功能混做。
- 先补自动化测试和健康检查，再升级依赖。
- 先在本机或临时环境验证构建、启动、核心 API 冒烟，再部署生产。
- 部署后观察 `journalctl -u diary-backend`，确认没有启动异常、参数绑定异常、缓存 key 异常和慢接口异常。
- 如果登录、写日记、相册、AI 报告任一核心路径失败，立即回滚到上一提交和上一份备份。
