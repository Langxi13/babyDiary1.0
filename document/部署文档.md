# 部署文档

本文使用 `/opt/baby-diary` 和 `diary.example.com` 作为示例。部署时应替换为自己的目录和域名，真实密钥只保存在服务器私有环境文件中。

## 部署结构

```text
/opt/baby-diary/
├── backend/
├── frontend/
├── config/application-prod.yml
├── data/images/
├── data/objects/
├── deploy/backend/
├── deploy/frontend/
├── scripts/
├── logs/
└── backups/
```

源码仓库不跟踪 `data/images/`、`deploy/` 产物、日志、备份、`.private/` 或任何环境文件。

## 环境要求

- Linux 和 systemd
- JDK 17、Maven 3.9+
- Node.js 22 LTS 和 npm
- MySQL 8.0+，推荐 8.4 LTS
- Docker，用于独立 Redis 容器
- Nginx
- 可选：FFmpeg/ffprobe、Tesseract OCR、支持中文的 PDF 字体
- Android 构建可选：JDK 21、Android SDK Platform 36、Build Tools 36
- iOS 构建可选：macOS、Xcode 26、Apple Developer 签名环境

前端使用 Vite 7，需要 Node.js `20.19+` 或 `22.12+`。推荐使用当前 Node.js 22 LTS。

## 私有配置

生产覆盖配置位于 `config/application-prod.yml`，敏感值从 `/etc/baby-diary/backend.env` 注入。该文件不属于仓库，权限必须设为 `600`：

```bash
install -d -m 700 /etc/baby-diary
install -m 600 /dev/null /etc/baby-diary/backend.env
```

环境文件示例：

```bash
DB_URL='jdbc:mysql://127.0.0.1:3306/baby-diary?connectionTimeZone=%2B08:00&forceConnectionTimeZoneToSession=true&useSSL=false&allowPublicKeyRetrieval=true'
DB_USERNAME=baby_diary_app
DB_PASSWORD=replace-with-a-strong-password
SERVER_ADDRESS=127.0.0.1
DIARY_FILE_PATH=/opt/baby-diary/data/images/
DIARY_OBJECT_PATH=/opt/baby-diary/data/objects/
DIARY_PAGE_SIZE=5
INVITATION_CODE=replace-with-a-private-code
INVITATION_CODE_ENCRYPTION_KEY=replace-with-at-least-32-random-characters
JWT_SECRET=replace-with-at-least-32-random-characters
JWT_EXPIRATION=2592000000
JWT_ACCESS_EXPIRATION=900000
AUTH_REFRESH_DAYS=30
AUTH_SECURE_COOKIE=true
AUTH_STEP_UP_MINUTES=10
PUBLIC_URL=https://diary.example.com
CORS_ALLOWED_ORIGINS=https://diary.example.com
CORS_NATIVE_ORIGINS=https://localhost,capacitor://localhost
REDIS_HOST=127.0.0.1
REDIS_PORT=6380
CACHE_PREFIX=baby-diary:prod:
AI_CONFIG_ENCRYPTION_KEY=replace-with-at-least-32-random-characters
OBJECT_STORAGE_PROVIDER=local
MEDIA_PROCESSING_ENABLED=true
FFMPEG_BIN=ffmpeg
FFPROBE_BIN=ffprobe
TESSERACT_BIN=tesseract
PDF_FONT_PATH=/usr/share/fonts/truetype/unifont/unifont.ttf
REMINDER_DELIVERY_DELAY_MS=60000
SLOW_REQUEST_THRESHOLD_MS=1000
SPRINGDOC_ENABLED=false
HEALTH_CHECK_BASE_URL=https://diary.example.com
HEALTH_CHECK_RESOLVE_HOST=diary.example.com:443:127.0.0.1
```

`JWT_SECRET`、`INVITATION_CODE_ENCRYPTION_KEY` 和 `AI_CONFIG_ENCRYPTION_KEY` 可通过 `openssl rand -base64 48` 生成。邀请码和 AI 配置使用不同密钥；任一加密密钥发生变化后，对应数据库密文都无法再解密。应用会在启动阶段验证邀请码密文，密钥不匹配时拒绝启动，禁止临时更换后直接发布。

`INVITATION_CODE` 只用于空数据库第一次启动：V14 表为空时，后端将其加密写入 `system_invitation_config`。初始化后注册只读取数据库，管理员可在个人信息页完成密码二次验证后查看和刷新。为了保留上一版本回滚能力，首次升级时先保留该变量；确认 V14、管理员查看和新用户注册正常后可移除，后续启动不再要求它。

生产后端默认且必须只监听 `127.0.0.1:10002`，由同机 Nginx 反向代理。不要把 `SERVER_ADDRESS` 改为 `0.0.0.0`，也不要在防火墙中公开 Java 端口；预发布 Compose 会单独覆盖为容器网络所需的 `0.0.0.0`。

认证默认值为：V2 访问令牌 15 分钟、旋转刷新会话 30 天、日记锁二次验证 10 分钟。`JWT_EXPIRATION` 的 30 天值只服务旧版兼容登录接口，新前端使用 V2 会话。生产环境必须使用 HTTPS 并保持 `AUTH_SECURE_COOKIE=true`。

`CORS_ALLOWED_ORIGINS` 只填写实际 Web 地址；`CORS_NATIVE_ORIGINS` 默认且建议保持 `https://localhost,capacitor://localhost`，分别对应 Android 与 iOS Capacitor WebView。两项都禁止使用 `*`。原生客户端连接的是用户在设备上输入并由兼容接口验证的 HTTPS 根地址，生产域名不应编译进 APK。

邮件发送使用 Spring Boot 标准变量。启用时至少配置：

```bash
MAIL_ENABLED=true
MAIL_FROM=diary@example.com
SPRING_MAIL_HOST=smtp.example.com
SPRING_MAIL_PORT=587
SPRING_MAIL_USERNAME=diary@example.com
SPRING_MAIL_PASSWORD=replace-with-an-app-password
SPRING_MAIL_PROPERTIES_MAIL_SMTP_AUTH=true
SPRING_MAIL_PROPERTIES_MAIL_SMTP_STARTTLS_ENABLE=true
```

Web Push 需要配置 `VAPID_PUBLIC_KEY`、`VAPID_PRIVATE_KEY` 和 `VAPID_SUBJECT`。S3 兼容存储需要把 `OBJECT_STORAGE_PROVIDER` 改为 `s3`，并配置 `S3_ENDPOINT`、`S3_REGION`、`S3_BUCKET`、`S3_ACCESS_KEY`、`S3_SECRET_KEY`。

## 构建与验证

```bash
cd /opt/baby-diary
npm --prefix frontend ci
scripts/verify.sh
scripts/verify-e2e.sh
scripts/security-scan.sh
```

`scripts/verify.sh` 会运行 Shell 脚本自测、后端 MySQL 8.4 集成测试与 JaCoCo 门禁、前端 Node/Vitest 测试与生产构建。发布候选还必须通过三浏览器 E2E、Trivy/Gitleaks、打包预发布 ZAP、k6 和受影响平台的真机验收。完整矩阵、阈值和回滚条件见 [测试与发布验收方案](测试与发布验收方案.md)。

预发布配置从 `config/staging.env.example` 创建为不提交仓库的 `.env.staging`，只写入随机密钥和合成数据：

```bash
scripts/start-staging.sh
scripts/run-zap-baseline.sh
scripts/run-k6.sh
scripts/stop-staging.sh
```

单独构建：

```bash
mvn -q -DskipTests clean package -f backend/pom.xml
npm --prefix frontend run build
```

Android Debug APK：

```bash
scripts/build-android.sh
```

Android 构建使用 JDK 21，脚本固定单 worker、无常驻 Gradle daemon，适合资源有限的服务器串行运行；不要与 Maven 全量测试或 Playwright 并发。Debug 产物位于 `frontend/android/app/build/outputs/apk/debug/app-debug.apk`。

正式签名材料通过 `scripts/ensure-android-signing.sh` 幂等初始化到仓库外的私有目录，环境文件和 JKS 均为 `600`，目录为 `700`。`scripts/sync-android-signing-secrets.sh` 在证书与 `config/android-release-cert.sha256` 一致后，才会通过标准输入写入 `ANDROID_KEYSTORE_BASE64`、`ANDROID_KEYSTORE_PASSWORD`、`ANDROID_KEY_ALIAS`、`ANDROID_KEY_PASSWORD` 四个 GitHub Secrets，秘密不会出现在进程参数或日志中。`scripts/backup.sh` 会把签名环境和 JKS 以 `600` 权限纳入校验备份；该备份还必须复制到服务器之外的离线安全介质。

正式版本集中维护在 `config/android-release-version.properties`。每次发布必须先提交递增的 `VERSION_CODE` 和新的 `VERSION_NAME`；本地使用 `scripts/build-android-release.sh` 读取该清单，依次执行前端构建、Capacitor 同步、Release lint、单元测试、APK/AAB 构建和 `scripts/verify-android-artifact.sh`。GitHub 手动工作流 `.github/workflows/android-release.yml` 从同一清单生成 `v<VERSION_NAME>` 标签，拒绝覆盖已有标签，并自动发布具名 APK、AAB 和校验和。Gradle Wrapper JAR 的 SHA-256 已与 Gradle 官方 8.14.3 校验值比对并纳入资产门禁。

Debug 与 Release 使用不同证书，首次安装签名 Beta 前必须卸载 Debug 包一次。后续 Release 保持同一证书并递增 `version_code`，即可覆盖升级；丢失永久签名密钥会导致现有安装无法再升级。

仅本机或模拟器联调可使用 `VITE_NATIVE_ALLOW_HTTP=true scripts/build-android.sh`；该包只放行 localhost、`10.0.2.2` 与私有局域网地址。生产签名工作流不得设置此变量，且 Release Manifest 固定 `usesCleartextTraffic=false`。

## systemd

建议使用专用系统用户运行后端：

```bash
useradd --system --home /opt/baby-diary --shell /usr/sbin/nologin baby-diary
```

`/etc/systemd/system/diary-backend.service` 示例：

```ini
[Unit]
Description=Baby Diary Backend Service
After=network.target mysql.service
Wants=mysql.service

[Service]
Type=simple
User=baby-diary
Group=baby-diary
WorkingDirectory=/opt/baby-diary
ExecStart=/usr/bin/java -Xms256m -Xmx1024m -jar /opt/baby-diary/deploy/backend/Baby-Diary-0.0.1-SNAPSHOT.jar --spring.config.location=file:/opt/baby-diary/config/application-prod.yml
EnvironmentFile=/etc/baby-diary/backend.env
SuccessExitStatus=143
Restart=on-failure
RestartSec=10
PrivateTmp=true
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
```

不要自定义基于 `kill` 的 `ExecStop`，让 systemd 使用默认 SIGTERM 关闭 Java 进程。仓库中的 `config/diary-backend-hardening.conf` 会由部署脚本安装为 systemd drop-in 并启用 `PrivateTmp=true`，使 Tomcat 临时文件与主机共享 `/tmp` 隔离。主机 `/tmp` 本身仍必须保持 `1777`，部署治理会同时验证权限和服务用户可写性。

## Redis

生产环境默认使用独立 Redis 容器并仅绑定回环地址：

```bash
cd /opt/baby-diary
scripts/ensure-redis.sh
```

默认配置为 `127.0.0.1:6380`、128MB 上限、`allkeys-lru` 和 AOF 持久化。可通过 `REDIS_MAXMEMORY`、`REDIS_MAXMEMORY_POLICY` 等环境变量覆盖。

后端已配置缓存错误降级。Redis 短暂不可用时读取会回源数据库，但应及时恢复 Redis，避免数据库长期承担全部读取流量。

## 存储目录

上传目录必须允许后端用户写入，并允许 Nginx 组读取：

```bash
cd /opt/baby-diary
SERVICE_USER=baby-diary NGINX_GROUP=www-data scripts/ensure-image-permissions.sh
```

脚本会把数据目录设为 `2750`、文件设为 `0640`，并使用 setgid 让新文件继承 Nginx 组。历史图片缩略图可通过以下命令回填：

```bash
scripts/generate-thumbnails.sh
```

同一脚本会将 `DIARY_OBJECT_PATH` 设置为仅后端用户可访问的目录模式 `0700`，对象文件为 `0600`。`DIARY_OBJECT_PATH` 不得位于 `DIARY_FILE_PATH` 之下，否则 V2 私有媒体可能被旧版静态图片路由直接读取，运行治理检查会拒绝部署。

## Nginx

以下配置只展示应用相关部分，TLS 证书和安全头应按实际环境配置：

```nginx
server {
    listen 443 ssl http2;
    server_name diary.example.com;
    server_tokens off;

    root /opt/baby-diary/deploy/frontend;
    client_max_body_size 300M;
    include /etc/nginx/snippets/baby-diary-security-headers.conf;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location ^~ /api/ {
        proxy_pass http://127.0.0.1:10002/api/;
        proxy_hide_header X-Content-Type-Options;
        proxy_hide_header X-Frame-Options;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    include /etc/nginx/snippets/baby-diary-backend-health.conf;

    location ^~ /images/ {
        include /etc/nginx/snippets/baby-diary-security-headers.conf;
        alias /opt/baby-diary/data/images/;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    location = /manifest.webmanifest {
        default_type application/manifest+json;
        try_files $uri =404;
    }
}
```

仓库中的 `config/nginx-security-headers.conf` 由部署脚本安装到 `/etc/nginx/snippets/baby-diary-security-headers.conf`。任何包含自定义 `add_header` 的 location 都会覆盖 server 级继承，因此 Manifest、SPA、静态资源和图片 location 也必须再次 include 该片段；部署治理检查会拒绝完全未引用该片段的站点配置。

`config/nginx-resource-policy-map.conf` 会安装到 `/etc/nginx/conf.d/baby-diary-resource-policy-map.conf`。它默认返回 `Cross-Origin-Resource-Policy: same-origin`，仅为 `/images/**` 和 `/api/v2/media/**` 返回 `cross-origin`，供 Capacitor WebView 显示远端图片。该 map 必须位于 Nginx `http` 上下文，不能放进单个 `server`；不要把默认值整体改成 `cross-origin`。

`config/nginx-backend-health.conf` 会安装到 `/etc/nginx/snippets/baby-diary-backend-health.conf`。站点 `server` 必须 include 该片段，它使用精确匹配把 `/actuator/health` 代理到回环地址上的后端，避免 SPA fallback 把后端故障伪装成 HTTP 200。不要把该路径改成前端静态响应或仅返回固定文本。

默认 CSP 只允许同源图片和媒体。启用 S3 兼容存储时，应把确定的对象存储源精确加入 `img-src`、`media-src`，必要时加入 `connect-src`，不要使用任意 `https:` 或 `*` 通配来源。

必须确保 `/manifest.webmanifest` 返回 `application/manifest+json`，否则部分移动浏览器无法正确识别 PWA 安装元数据、图标和启动方式。

`/images/**` 仅为旧版图片兼容入口，仍是公开静态资源。V2 图片、音频和视频必须保存在独立的 `DIARY_OBJECT_PATH` 或 S3 中，并通过 `/api/v2/media/public/**` 的短时签名 URL 读取。不要把对象目录配置为 Nginx `alias`。

## 部署

首次部署前创建运行目录：

```bash
install -d /opt/baby-diary/deploy/backend /opt/baby-diary/deploy/frontend
```

统一部署流程：

```bash
cd /opt/baby-diary
scripts/backup.sh
scripts/deploy.sh
```

部署脚本会构建前后端、确保 AI 与邀请码加密密钥和 Redis 可用、隔离旧图片与私有对象目录、生成缩略图、安装 Nginx 安全头、原生资源策略和 systemd 治理片段并执行运行治理检查。缺少 `INVITATION_CODE_ENCRYPTION_KEY` 时，脚本只在权限为 `600` 的生产环境文件中生成一次，不会输出密钥。脚本会在停止后端前运行 `nginx -t`，随后按停止服务、复制 Jar、同步前端、启动服务、重载 Nginx、健康检查的顺序发布。

不要在 Java 服务运行时直接覆盖正在执行的 Jar。

## 健康检查

`scripts/health-check.sh` 从 `/etc/baby-diary/backend.env` 读取 `HEALTH_CHECK_BASE_URL` 和可选的 `HEALTH_CHECK_RESOLVE_HOST`，也允许临时使用 `BASE_URL`、`RESOLVE_HOST` 覆盖：

```bash
scripts/health-check.sh
```

健康检查同时验证 SPA 页面、未登录认证响应、`/api/v2/client/bootstrap` 原生兼容入口、PWA Manifest 类型和 Actuator 顶层 `UP` 状态。

默认检查首页、相册、日记列表、Actuator、未登录认证响应和 PWA manifest 类型。Actuator 不只要求 HTTP 200 和 JSON 类型，还要求响应顶层为 `{"status":"UP"}`；返回 SPA HTML 或 `DOWN` 都会阻断发布。后端仅暴露 Actuator `health` 端点。

## 数据库迁移

后端启动时由 Flyway 自动执行 `backend/src/main/resources/db/migration/` 下尚未应用的迁移。当前结构版本为 V14：V9 建立空间与身份基础，V10 增加协作、同步和通知，V11 增加搜索、模板、富媒体和私密分享，V12 增加提醒投递保护，V13 统一相册封面与日记图片路径的精确匹配排序规则，V14 增加加密的系统邀请码配置。发布前应先备份数据库，不要修改已经在生产环境执行过的迁移文件。

`baseline-on-migrate` 只用于结构与 V1 一致但尚无 Flyway 历史表的旧实例。如果数据库已经包含后续版本字段却没有 `flyway_schema_history`，不要直接启动应用，应先人工核对并建立正确基线，否则可能出现重复字段迁移失败。

应用数据库账号应是只服务于本项目的最小权限账号，不应使用 MySQL `root`。

## 备份与恢复

```bash
cd /opt/baby-diary
scripts/backup.sh
scripts/verify-backup.sh backups/20260101-120000
```

备份包含项目归档、数据库 dump、校验和以及存在时的 Nginx、systemd 和私有环境配置。备份本身包含敏感数据，应加密存储、限制访问且不得提交到 Git。

恢复前先停止后端，恢复数据库、图片目录、环境配置和部署产物，再执行：

```bash
systemctl daemon-reload
systemctl restart diary-backend
nginx -t
systemctl reload nginx
scripts/health-check.sh
```
